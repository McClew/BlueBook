# Operational Context Questionnaire (OCQ)

## Overview & Definition

### What is an OCQ?

An Operational Context Questionnaire (OCQ) - also frequently referred to as an _Environmental Baseline Document_ or _Business Logic Mapping Form -_ is a structured framework used by Security Operations Centres (SOCs) during the onboarding and engineering phases of a client relationship.

Unlike standard technical checklists (which focus on log sources, IP addresses, and credential management), an OCQ captures the business logic, operational norms, and risk tolerances of a target environment. It translates organisational realities into actionable data that engineers can feed directly into SIEM, EDR, and CDR detection pipelines.

```
+----------------------------+     +-------------------------------+     +----------------------------+
|    Technical Ingestion     |  +  |      Business Context         |  =  |   High-Fidelity Triage     |
| (Logs, IPs, API Endpoints) |     | (OCQ: Allowed tools, roles)   |     | (Low False-Positive Rate)  |
+----------------------------+     +-------------------------------+     +----------------------------+
```

### Why Use It?

Defending an unfamiliar network without business context forces Tier 1 and Tier 2 analysts to operate in a vacuum. This knowledge gap introduces two critical systemic failure points:

* Alert Fatigue (The "Noise" Problem): Analysts spend valuable cycles triaging benign-positive alerts generated by authorized but anomalous activities (e.g., developers testing code, IT admins using remote access tools).
* Blind Spots (The "False-Negative" Problem): To mitigate noise, engineers frequently implement over-broad, global suppression rules. This allows sophisticated adversaries to execute living-off-the-land attacks using the exact same tools and techniques as the internal IT staff without triggering an alarm.

By formalising the OCQ process, the Blue Team shifts from a posture of reactive, statistical anomaly detection to highly tailored, context-aware threat monitoring.

## Core Components & Expanded Scope

A comprehensive OCQ must span multiple technological and organisational domains. To scale across a diverse client portfolio, the questionnaire is structured into distinct thematic modules:

### Identity, Collaboration & SaaS Governance

This module maps how an enterprise manages its cloud productivity suites (e.g., Microsoft 365, Google Workspace) and identity providers (IdPs).

* **External & Anonymous Sharing Policies:** Are users permitted to generate unauthenticated, anonymous sharing links for cloud storage? Are there restricted or explicitly authorised external domains/tenants for B2B collaboration?
* **Administrative Geographies & Travel Profiles:** Where do cloud administrators typically log in from? Is conditional access configured to block non-domestic traffic, or must the SOC differentiate between legitimate international travel and session-hijacking/adversary-in-the-middle (AiTM) attacks?
* **Low-Code/No-Code Automation Footprint:** Is the broader user base permitted to deploy automated workflows (e.g., Power Automate, Zapier)? What are the designated egress pathways for enterprise data via these integrations?

### Host-Level Behaviour & Administrative Tooling

This module establishes what constitutes authorised "admin behavior" across workstations, servers, and cloud workloads.

* **Developer, DevOps & Engineering Baselines:** Which specific user cohorts or active directory groups are authorised to invoke compilers, interpreters, and native shells (e.g., Python, Git, PowerShell, Bash, `cmd.exe`)?
* **Remote Monitoring & Management (RMM) Matrix:** What tools are officially sanctioned for internal IT administration? If a client uses _ScreenConnect_, the SOC must be primed to treat _AnyDesk_ or _TeamViewer_ executions as immediate indicators of compromise (IoCs).
* **Legacy & Bespoke Application Registry:** Does the enterprise rely on custom-built, unsigned, or legacy software that exhibits malware-like characteristics (e.g., process injection, spawning shells, or writing executable files directly into `%AppData%`)?

### Network Architecture & Egress Anomalies

This module defines the boundaries of the network perimeter and identifies expected data movement.

* **Sanctioned Cloud Storage & Exfiltration Vectors:** Which cloud backup or storage providers are corporate-approved (e.g., OneDrive, corporate Dropbox)? This allows analysts to immediately escalate data transfers occurring via unsanctioned tools like MegaSync or Cyberduck.
* **Dual-Homed Systems & Shadow IT Infra:** Are there known systems operating outside the standard perimeter (e.g., standalone lab environments, manufacturing/OT jumpboxes) that naturally communicate over anomalous ports?
* **Inbound/Outbound Threat Intel Feed Alignment:** Are there specific geographic regions or dynamic hosting providers (e.g., specific AWS/Azure regions) with which the client maintains a necessary, high-volume business relationship?

### Organisational & Operational Context

This module captures the human elements of the business that influence risk management decisions.

* **VIP & High-Value Target (HVT) Registry:** Who are the key executives, finance officers, and domain administrators? Alerts affecting these identities must bypass standard triage queues and automatically escalate to priority status.
* **Change Management Windows & Maintenance Cycles:** When do internal IT and DevOps teams perform disruptive maintenance, patching, or network scans? Correlating high-volume configuration changes with scheduled windows prevents midnight panic calls for routine operations.
* **Third-Party Risk & Vendor Integration:** Which external vendors hold persistent, privileged access into the client environment? What tools (e.g., Site-to-Site VPNs, privileged access management portals) do they use to connect?

## Engineering Implementation

Data collected via the OCQ must not sit dormant in a document repository. It must be actively weaponised to tune security controls and detection logic.

### Watchlists and Lookup Tables

Instead of hard-coding client exceptions directly into SIEM or EDR detection rules - which creates unmanageable code sprawl - onboarding engineers must map OCQ responses to dynamic lookup tables.

> **Analyst Workflow Example:**
>
> When an EDR alert fires for a potentially malicious administrative tool execution (`PsExec.exe`), the detection engine parses a lookup table populated by the OCQ data:
>
> \$$\text{If } \textit{Executing\\\_User} \in \text{Lookup}(\text{"Approved\\\_IT\\\_Admins"}) \rightarrow \text{Set Severity to Low / Suppress}\$$\$$\text{If } \textit{Executing\\\_User} \notin \text{Lookup}(\text{"Approved\\\_IT\\\_Admins"}) \rightarrow \text{Escalate to Critical Incident}\$$

### The Exception Sign-Off Matrix

If a client confirms during the OCQ process that they accept an inherently high-risk behavior (e.g., allowing anonymous file sharing or disabling MFA for legacy service accounts), this decision must be documented within a Sign-Off Matrix.

This matrix serves two purposes:

1. It creates clear operational boundaries, allowing SOC analysts to tag known-risky but business-sanctioned events as Benign-Positive rather than True-Positive threats.
2. It protects the provider from liability by creating a documented paper trail of accepted risk signed by the client’s CISO or Risk Officer.

### Lifecycle & Recertification

Business operations are fluid. A baseline captured on day one will inevitably decay over time. To prevent context drift, the Blue Team should institute a mandatory Quarterly or Bi-Annual OCQ Recertification. During regular business reviews, analysts present current lookup tables to the client to verify if authorised groups, tools, and habits require updating.
